# 资产模块生成提示词

设计一个【资产模块三页原型】，用于【统一查看设备信息与检修员信息，并分别进入设备详情页、检修员详情页进行高密度二维表管理】。

本提示词的核心约束是：资产模块一共三页，不是两页，也不是一个页面加常驻抽屉。

==================================================

【一、页面定位（Page Role）】

- 页面类型：卡片式资产工作台 + 两个高密度二维表详情页（三页组合）
- 用户角色：设备与人员管理人员、调度指挥人员、检修负责人
- 三个页面固定为：
  1. 页面A：资产工作台首页
     - 同页包含设备信息区和检修员信息区
     - 两个容器上下排布
     - 每个容器内部固定三行卡片
     - 页面使用普通纵向滚动
  2. 页面B：设备详情页
     - 使用高密度二维表展示设备信息
     - 用于承接设备信息区的“查看全部”或设备详情入口
  3. 页面C：检修员详情页
     - 使用高密度二维表展示检修员信息
     - 用于承接检修员信息区的“查看全部”或检修员详情入口
- 核心目标：
  - 目标1：首页用两个受控容器展示重点设备和重点检修员，不把全部数据塞进首页
  - 目标2：设备详情页和检修员详情页分别用高密度二维表承接全量数据
  - 目标3：抽屉只作为点击后的临时详情层，不是页面常态显示结构
- 使用场景：管理人员日常查看设备状态、检修员资源分布，快速进入对应详情页进行筛选、检索和表格管理

------

【二、整体布局（Layout System）】

本模块包含三个页面层级：

**页面A：资产工作台首页**

- 顶部结构：固定主导航 + 一键跳转区（设备信息 / 检修员信息快速定位）
- 主体布局：单栏上下排布，普通纵向滚动，非 Fullpage，非 Scroll Snap
- 区域划分：
  - 上方容器：设备信息区
  - 下方容器：检修员信息区
  - 一键跳转区：用于在两个容器之间快速定位
- 容器规则：
  - 首页只有两个大容器
  - 两个容器必须上下排布，不做左右并列
  - 每个容器内部固定三行卡片
  - 每行可放 2-4 张卡片，具体数量根据屏幕宽度自适应
  - 每个容器末尾提供“查看全部”入口
- 滚动规则：
  - 页面A使用普通页面滚动
  - 不限制为单屏
  - 不使用全屏分屏滚动

**页面B：设备详情页**

- 顶部结构：固定主导航 + 面包屑（资产 / 设备详情）+ 筛选工作区
- 主体布局：高密度二维表
- 区域划分：
  - 顶部：设备统计摘要与筛选栏
  - 中部：设备信息二维表
  - 右侧抽屉：点击表格行或详情按钮后临时显示
- 是否使用特殊布局：Data Grid，高密度表格优先

**页面C：检修员详情页**

- 顶部结构：固定主导航 + 面包屑（资产 / 检修员详情）+ 筛选工作区
- 主体布局：高密度二维表
- 区域划分：
  - 顶部：检修员统计摘要与筛选栏
  - 中部：检修员信息二维表
  - 右侧抽屉：点击表格行或详情按钮后临时显示
- 是否使用特殊布局：Data Grid，高密度表格优先

------

【三、信息架构（Information Architecture）】

模块1：设备信息区（页面A，上方容器）

- 功能说明：展示设备状态摘要与重点设备关注信息
- 包含信息：设备卡片、运行状态、关键参数、所属区域、所属管道、最近巡检时间
- 展示规则：
  - 固定三行卡片
  - 不展示全量设备
  - 超出三行的信息通过“查看全部设备”进入页面B
- 优先级：高

模块2：检修员信息区（页面A，下方容器）

- 功能说明：展示检修员资源分布与值守状态
- 包含信息：检修员卡片、在岗状态、负责区域、负责设备、当前任务数、联系方式摘要
- 展示规则：
  - 固定三行卡片
  - 不展示全量检修员
  - 超出三行的信息通过“查看全部检修员”进入页面C
- 优先级：高

模块3：设备详情表（页面B）

- 功能说明：承接设备全量数据与设备筛选、检索、排序、查看详情
- 包含信息：高密度二维表
- 建议字段：
  - 设备编号
  - 设备名称
  - 设备类型
  - 所属区域
  - 所属管道
  - 运行状态
  - 压力/流量/温度等关键参数摘要
  - 最近巡检时间
  - 负责人
  - 操作列
- 优先级：高

模块4：检修员详情表（页面C）

- 功能说明：承接检修员全量数据与人员筛选、检索、排序、查看详情
- 包含信息：高密度二维表
- 建议字段：
  - 工号
  - 姓名
  - 所属班组
  - 负责区域
  - 负责设备数
  - 当前任务数
  - 在岗状态
  - 联系方式
  - 最近处理时间
  - 操作列
- 优先级：高

模块5：点击后详情抽屉（页面A/B/C均可使用）

- 功能说明：展示单个设备或单个检修员的补充信息
- 显示规则：
  - 抽屉默认不显示
  - 抽屉不是常态页面布局的一部分
  - 只有点击卡片、表格行或“详情”按钮后才从右侧滑入
  - 关闭后页面恢复原始布局
- 内容规则：
  - 设备抽屉展示设备基础信息、关键状态、最近记录、关联检修员
  - 检修员抽屉展示人员基础信息、负责区域、当前任务、最近记录
  - 不在抽屉里承载全量表格

------

【四、核心模块详细设计（Key Module Design）】

模块名称：设备信息区（首页）

- 展示形式：大容器 + 三行设备卡片
- 每个设备卡片包含：
  - 设备名称 / 设备编号（使用等宽字体突出编号）
  - 运行状态标识
  - 所属区域 / 所属管道
  - 关键参数摘要
  - 最近巡检时间
- 操作区：
  - 主操作：点击卡片 → 打开右侧临时抽屉
  - 次操作：“查看全部设备” → 跳转设备详情页
- 状态设计：
  - 正常运行：青色状态
  - 需关注：黄色状态
  - 故障/离线：橙色或错误状态

模块名称：检修员信息区（首页）

- 展示形式：大容器 + 三行检修员卡片
- 每个检修员卡片包含：
  - 检修员姓名 / 工号
  - 在岗状态
  - 所属班组
  - 负责区域 / 负责设备
  - 当前任务数
- 操作区：
  - 主操作：点击卡片 → 打开右侧临时抽屉
  - 次操作：“查看全部检修员” → 跳转检修员详情页
- 状态设计：
  - 在岗：青色或绿色状态
  - 任务中：黄色状态
  - 离岗/不可用：弱化状态

模块名称：设备详情页二维表

- 展示形式：高密度 Data Grid
- 表格能力：
  - 筛选
  - 搜索
  - 排序
  - 分页
  - 列显示控制
  - 导出（如果现有接口或前端能力允许）
- 行操作：
  - 点击行选中
  - 双击或点击详情按钮 → 打开右侧临时抽屉
  - 不跳转到第四个页面

模块名称：检修员详情页二维表

- 展示形式：高密度 Data Grid
- 表格能力：
  - 筛选
  - 搜索
  - 排序
  - 分页
  - 列显示控制
  - 导出（如果现有接口或前端能力允许）
- 行操作：
  - 点击行选中
  - 双击或点击详情按钮 → 打开右侧临时抽屉
  - 不跳转到第四个页面

------

【五、交互模型（Interaction Model）】

- 主交互方式：
  - 首页：卡片浏览 + 快速定位 + 查看全部
  - 设备详情页：设备二维表浏览 + 筛选控制
  - 检修员详情页：检修员二维表浏览 + 筛选控制
- 分层交互：
  - 第一层：首页两个容器浏览（设备信息区、检修员信息区）
  - 第二层：点击“查看全部设备”或“查看全部检修员”进入对应详情页
  - 第三层：在详情页中使用高密度表格筛选、排序、查看
  - 第四层：点击卡片或表格行后临时打开右侧抽屉
- 页面跳转策略：
  - 首页 → 设备详情页：点击设备信息区的“查看全部设备”
  - 首页 → 检修员详情页：点击检修员信息区的“查看全部检修员”
  - 设备详情页 / 检修员详情页 → 首页：通过面包屑或返回入口
  - 抽屉不产生新路由，不作为常态页面
- 动态行为：
  - 卡片 Hover 高亮
  - 表格行 Hover 高亮
  - 抽屉点击后滑入，关闭后滑出
  - 筛选应用后表格刷新

------

【六、视觉与层级（Visual Hierarchy）】

- 视觉优先级：
  1. 首页设备信息区（核心资产）
  2. 首页检修员信息区（人力资源）
  3. 设备详情页 / 检修员详情页的高密度二维表
  4. 点击后出现的临时抽屉
- 设计风格：Tactical Brutalism — 冷硬监控台语言，交互以高效管理为主，少做装饰
- 组件风格：
  - 大容器：0px 圆角，明确边界，上下排布
  - 卡片：0px 圆角，紧凑信息块，状态色只用于关键标识
  - 表格：高密度二维表，细边框分隔，紧凑行高
  - 抽屉：右侧覆盖层，点击后出现，默认不占布局空间
- 状态颜色规范：
  - 正常 / 在岗：青色或绿色
  - 需关注 / 任务中：黄色
  - 故障 / 高风险：橙色或错误色

------

【七、动效设计（Motion & Feedback）】

- Hover 动效：卡片和表格行背景轻微变化
- 切换动画：抽屉滑入（Mechanical Step-easing）
- 加载动画：表格刷新时使用克制的扫描线或进度条
- 操作反馈：抽屉打开/关闭、筛选应用、导出完成要有明确反馈

------

【八、扩展能力（Advanced Features，可选）】

- 首页一键跳转区快速定位设备信息区 / 检修员信息区
- 重点设备关注区（展示层聚合）
- 检修资源分布可视化
- 设备详情页支持保存筛选视图
- 检修员详情页支持按班组、区域、状态快速筛选

------

【九、API 契约（必须使用的具体接口）】

所有页面必须基于以下已有接口生成原型或实现，不允许虚构新的后端接口。

通用认证约束：

- 除区域三级联动接口外，业务接口均需要携带登录态：
  - `Authorization: Bearer <token>`
- 统一响应结构遵循：
  - 成功：`{ code: 200, message: string, data: ... }`
  - 分页：`{ code: 200, message: string, data: { records, total, page, pageSize } }`

页面A：资产工作台首页

- 设备信息区摘要与三行设备卡片：
  - `GET /data_management/equipment/Indicator_card`
    - 用途：获取设备指标卡数据，用于设备信息区顶部摘要
  - `POST /data_management/equipment/find_equipment_params`
    - 用途：获取首页设备卡片数据
    - 首页只取有限数量，例如 `page=1`、`page_size=9` 或按三行卡片容量取数
    - 不允许为了填满首页调用全量数据后前端无限堆卡片
- 检修员信息区摘要与三行检修员卡片：
  - `GET /data_management/repairman/Indicator_card`
    - 用途：获取检修员指标卡数据，用于检修员信息区顶部摘要
  - `POST /data_management/repairman/find_repairman_params`
    - 用途：获取首页检修员卡片数据
    - 首页只取有限数量，例如 `page=1`、`page_size=9` 或按三行卡片容量取数
- 区域筛选/展示名称辅助接口：
  - `GET /area_details/provinces`
  - `GET /area_details/citys/{province_code}`
  - `GET /area_details/districts/{city_code}`
  - 用途：区域筛选、区域名称显示、区域联动选择

页面B：设备详情页

- 设备高密度二维表：
  - `POST /data_management/equipment/find_equipment_params`
    - 用途：设备详情页主表格查询
    - 支持分页、搜索、筛选、排序展示
    - 使用文档已有参数，不新增后端未提供筛选项
- 设备行详情抽屉：
  - `POST /data_management/equipment/find_equipment_id`
    - 用途：点击设备卡片、表格行或详情按钮后，获取单个设备详情并打开右侧临时抽屉
    - 请求体使用设备 `id`
- 设备新增、修改、删除如果页面原型需要展示操作入口，只能使用：
  - `POST /data_management/equipment/add_equipment`
  - `POST /data_management/equipment/update_equipment`
  - `GET /data_management/equipment/delete_equipment/{ids}`
  - 如果当前原型只做浏览和详情，不要强行展示新增/修改/删除流程

页面C：检修员详情页

- 检修员高密度二维表：
  - `POST /data_management/repairman/find_repairman_params`
    - 用途：检修员详情页主表格查询
    - 支持分页、姓名模糊查询、性别、年龄范围、区域、入职时间、电话等文档已有筛选项
    - 不新增后端未提供筛选项
- 检修员行详情抽屉：
  - `POST /data_management/repairman/find_repairman_id`
    - 用途：点击检修员卡片、表格行或详情按钮后，获取单个检修员详情并打开右侧临时抽屉
    - 请求体使用检修员 `id`
- 检修员新增、修改、删除如果页面原型需要展示操作入口，只能使用：
  - `POST /data_management/repairman/add_repairman`
  - `POST /data_management/repairman/update_repairman`
  - `GET /data_management/repairman/delete_repairman/{ids}`
  - 如果当前原型只做浏览和详情，不要强行展示新增/修改/删除流程

接口与页面映射限制：

- 首页设备容器不得调用检修员接口填充设备卡片
- 首页检修员容器不得调用设备接口填充检修员卡片
- 设备详情页只承接设备接口
- 检修员详情页只承接检修员接口
- 抽屉详情必须使用对应 ID 查询接口，不用列表接口结果假装详情
- 不允许新增“资产总览统一查询接口”“设备检修员混合查询接口”等后端不存在的接口

------

【十、设计目标（Design Goals）】

- 明确三页结构，避免把设备与检修员全量数据混在同一张详情表中
- 首页只做有限展示：两个容器上下排布，每个容器三行卡片
- 首页使用普通滚动，允许用户自然浏览两个容器
- 设备详情页和检修员详情页都使用高密度二维表承接全量数据
- 抽屉只在点击后显示，不作为常态页面结构
- 提升资产检索、状态感知和人员调度效率

------

【十一、限制与规范（Constraints）】

- 业务来源：原项目“数据管理”中的设备信息模块 + 检修员信息模块
- 本模块固定三页：
  - 资产工作台首页
  - 设备详情页
  - 检修员详情页
- 不允许生成“两页组合”方案
- 不允许把设备和检修员全量数据合并成一个通用全量数据页
- 首页不能把所有卡片无节制塞满
- 首页两个大容器必须上下排布
- 首页每个容器内部固定三行卡片
- 首页使用普通纵向滚动，不使用 Fullpage 或 Scroll Snap
- 设备详情页必须使用高密度二维表
- 检修员详情页必须使用高密度二维表
- 抽屉默认不显示，只能由点击卡片、表格行或详情按钮触发
- 抽屉不替代设备详情页或检修员详情页
- 所有功能必须基于现有设备管理和检修员管理接口
- 具体接口必须遵守本提示词“API 契约”章节
- 不使用圆角（borderRadius: 0px）
- 不使用纯灰色（#808080）
- 不为了交互完整而凭空增加后端没有的操作流
