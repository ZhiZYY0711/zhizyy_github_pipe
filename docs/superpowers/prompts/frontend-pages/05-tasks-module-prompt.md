# 任务模块生成提示词

设计一个【任务模块两页原型】，用于【管理巡检、复核、派发和闭环任务，并通过图表分析、重点任务突出展示、高密度二维表完成任务调度】。

本提示词的核心约束是：页面A不是纯卡片页，必须同时包含图表、KPI榜单、重点任务卡片和可展开筛选框；页面B是高密度二维表，并且页面A的重点任务卡片与页面B表格使用同一个任务列表接口。

==================================================

【一、页面定位（Page Role）】

- 页面类型：任务分析工作台 + 高密度二维表（双页组合）
- 用户角色：任务调度人员、执行跟踪人员、值班负责人
- 两个页面固定为：
  1. 页面A：任务工作台首页
     - 左侧展示重点任务卡片，只展示少量重点任务
     - 卡片点击后弹出模态框展示任务详情
     - 左侧区域提供“所有任务”按钮，点击后跳转页面B
     - 右侧展示三个图表和 KPI 榜单
     - 右上角绝对定位筛选框，可展开、可压缩
  2. 页面B：所有任务列表页
     - 使用高密度二维表展示任务全量数据
     - 与页面A重点任务卡片使用同一个列表接口
     - 点击行或详情按钮后弹出模态框展示任务详情
- 核心目标：
  - 目标1：页面A用于快速判断任务态势，不只做卡片堆叠
  - 目标2：重点任务优先露出，但不在首页展示所有任务
  - 目标3：页面B承接所有任务的筛选、分页、排序、表格浏览
  - 目标4：详情使用模态框，不使用常驻抽屉，不新增独立详情页

------

【二、整体布局（Layout System）】

本模块包含两个页面层级：

**页面A：任务工作台首页**

- 顶部结构：固定主导航
- 主体布局：左右分区
  - 左侧：重点任务卡片区
  - 右侧：图表分析区 + KPI 榜单
- 右上角筛选框：
  - 使用绝对定位
  - 默认可压缩成一个小型筛选按钮/筛选条
  - 点击后展开为筛选面板
  - 不挤压主布局，不改变左右区域宽度
- 左侧重点任务区域：
  - 只展示几个重点任务卡片，建议 4-8 个
  - 任务来源与页面B相同：`POST /data_management/tasks/find_task_params`
  - 可通过参数筛选高优先级、待处理、进行中或逾期任务
  - 区域内必须有“所有任务”按钮，跳转页面B
- 右侧图表区域：
  - 图表1：任务状态分布饼图
  - 图表2：任务数量趋势折线图
  - 图表3：区域任务对比柱状图
  - KPI榜单：检修员 KPI 排行榜
- 是否使用特殊布局：任务态势分析工作台，不做单纯卡片流，不做营销式大屏

**页面B：所有任务列表页**

- 顶部结构：固定主导航 + 面包屑（任务 / 所有任务）+ 筛选工作区
- 主体布局：高密度二维表
- 区域划分：
  - 顶部：任务统计摘要与筛选栏
  - 中部：任务二维表
  - 详情层：点击表格行或详情按钮后弹出模态框
- 是否使用特殊布局：Data Grid，高密度表格优先

------

【三、信息架构（Information Architecture）】

模块1：重点任务卡片区（页面A左侧）

- 功能说明：突出当前最需要处理或关注的任务
- 包含信息：任务标题、任务编号、优先级、状态、区域、管道、负责人、发布时间、响应时间或截止时间
- 展示规则：
  - 只展示少量重点任务
  - 不把所有任务卡片塞进页面A
  - 重点任务来源必须和页面B相同，使用任务列表接口筛选得到
- 操作：
  - 点击任务卡片 → 弹出模态框展示任务详情
  - 点击“所有任务” → 跳转页面B
- 优先级：最高

模块2：右上角筛选框（页面A）

- 功能说明：控制页面A图表和重点任务卡片的数据范围
- 展示规则：
  - 绝对定位在页面右上角
  - 可展开，可压缩
  - 压缩状态只显示筛选摘要，例如“华北 / 近7天 / 进行中”
  - 展开状态显示区域、任务类型、优先级、状态、时间范围等筛选项
- 交互规则：
  - 应用筛选后同时刷新左侧重点任务、右侧三个图表和 KPI 榜单
  - 不占据主内容流空间
- 优先级：高

模块3：任务状态分布饼图（页面A右侧）

- 功能说明：展示不同任务状态数量分布
- 数据来源：`POST /data_visualization/task_details/task_status`
- 展示形态：饼图或环形图
- 优先级：高

模块4：任务数量趋势折线图（页面A右侧）

- 功能说明：展示任务数量随时间变化趋势
- 数据来源：`POST /data_visualization/task_details/task_nums`
- 展示形态：折线图
- 优先级：高

模块5：区域任务对比柱状图（页面A右侧）

- 功能说明：展示不同区域任务数量对比
- 数据来源：`POST /data_visualization/task_details/task_contrast`
- 展示形态：柱状图
- 优先级：中高

模块6：KPI榜单（页面A右侧）

- 功能说明：展示检修员任务完成、响应时间或满意度排行
- 数据来源：`POST /data_visualization/task_details/KPI_list`
- 展示形态：榜单 / 排名列表
- 优先级：中高

模块7：所有任务二维表（页面B）

- 功能说明：承接所有任务的分页、筛选、搜索、排序和详情查看
- 数据来源：`POST /data_management/tasks/find_task_params`
- 包含信息：高密度二维表
- 建议字段：
  - 任务ID
  - 任务标题
  - 任务类型
  - 区域
  - 管道
  - 优先级
  - 状态
  - 指派检修员
  - 联系方式
  - 发布时间
  - 响应时间
  - 完成时间
  - 反馈信息
  - 操作列
- 优先级：高

模块8：任务详情模态框（页面A/B均可使用）

- 功能说明：展示单个任务详情
- 数据来源：`POST /data_management/tasks/find_task_id`
- 显示规则：
  - 默认不显示
  - 点击页面A重点任务卡片、页面B表格行或详情按钮后弹出
  - 模态框关闭后返回原页面
  - 不使用右侧常驻抽屉
  - 不新增独立详情页

------

【四、核心模块详细设计（Key Module Design）】

模块名称：页面A重点任务卡片区

- 展示形式：左侧纵向重点任务列表 / 卡片组
- 每张任务卡片包含：
  - 任务标题 / 任务编号
  - 状态标识：待处理 / 进行中 / 已完成 / 已取消
  - 优先级标识：紧急 / 高 / 中 / 低
  - 区域与管道
  - 指派检修员与联系方式摘要
  - 发布时间、响应时间或截止时间
- 数据策略：
  - 使用 `POST /data_management/tasks/find_task_params`
  - 通过 `priority`、`status`、`area_id`、`start_time`、`end_time` 等已有参数筛选重点任务
  - `page_size` 控制在少量展示，不做无限卡片流
- 操作区：
  - 点击卡片 → 调用 `POST /data_management/tasks/find_task_id` → 弹出详情模态框
  - “所有任务”按钮 → 跳转页面B

模块名称：页面A图表分析区

- 展示形式：右侧图表网格
- 必须包含：
  1. 任务状态分布饼图
  2. 任务数量趋势折线图
  3. 区域任务对比柱状图
  4. KPI榜单
- 数据策略：
  - 四个分析模块全部来自 `docs/api/02-数据可视化模块.md` 的模块2：任务详情分析
  - 不使用任务列表接口手工拼装图表数据，除非接口返回缺失且明确标记为临时展示层推导
- 视觉规则：
  - 图表服务决策效率，避免大屏装饰化
  - 数据线、柱、扇区必须使用设计系统语义色

模块名称：页面A右上角筛选框

- 展示形式：绝对定位浮层
- 状态：
  - 压缩状态：展示当前筛选摘要和展开按钮
  - 展开状态：展示完整筛选表单
- 筛选项：
  - 区域 `area_id`
  - 任务类型 `type`
  - 优先级 `priority`
  - 状态 `status`
  - 时间范围 `start_time` / `end_time`
- 行为：
  - 应用筛选时，刷新左侧重点任务和右侧图表
  - 重置筛选时恢复默认任务态势视图

模块名称：页面B高密度二维表

- 展示形式：Data Grid
- 数据来源：
  - 与页面A重点任务卡片相同：`POST /data_management/tasks/find_task_params`
- 表格能力：
  - 分页
  - 筛选
  - 搜索
  - 排序
  - 列显示控制
  - 导出（如果前端或现有接口允许）
- 行操作：
  - 点击行选中
  - 双击或点击详情按钮 → 调用 `POST /data_management/tasks/find_task_id` → 弹出详情模态框

------

【五、交互模型（Interaction Model）】

- 主交互方式：
  - 页面A：重点任务卡片 + 图表分析 + KPI榜单 + 右上角可展开筛选框
  - 页面B：高密度表格浏览 + 筛选控制 + 模态框详情
- 分层交互：
  - 第一层：页面A查看任务态势与重点任务
  - 第二层：点击重点任务卡片打开详情模态框
  - 第三层：点击“所有任务”进入页面B
  - 第四层：页面B中使用高密度二维表进行全量任务管理
- 页面跳转策略：
  - 页面A → 页面B：点击左侧重点任务区域的“所有任务”按钮
  - 页面B → 页面A：通过面包屑或返回入口
  - 任务详情不跳转，统一使用模态框
- 动态行为：
  - 筛选框展开/压缩
  - 图表刷新
  - 重点任务卡片 Hover 高亮
  - 逾期或紧急任务可轻微脉冲提示
  - 模态框打开/关闭即时反馈

------

【六、视觉与层级（Visual Hierarchy）】

- 视觉优先级：
  1. 页面A左侧重点任务卡片
  2. 页面A右侧图表中的任务状态分布与趋势
  3. 页面A右侧 KPI 榜单
  4. 页面A右上角筛选框
  5. 页面B高密度任务二维表
- 设计风格：Tactical Brutalism — 冷硬监控台，突出快速决策与高效操作
- 组件风格：
  - 卡片：0px 圆角，紧凑信息块
  - 图表：深色面板、细网格线、高对比数据线
  - 筛选框：绝对定位浮层，压缩态克制，展开态清晰
  - 模态框：居中或偏右覆盖层，不常驻，不挤压主布局
  - 全量列表：高密度 Data Grid
- 状态颜色规范：
  - 待处理：青色
  - 进行中：橙色
  - 逾期 / 紧急：黄色或橙色高亮
  - 已完成：低强调稳定色

------

【七、动效设计（Motion & Feedback）】

- Hover 动效：卡片、表格行、榜单行背景轻微变化
- 筛选框动画：展开/压缩使用短促机械感过渡
- 图表刷新：淡入更新，不做夸张动画
- 模态框：点击卡片或表格行后打开，关闭后回到原布局
- 操作反馈：筛选应用、任务状态更新、导出完成需要有明确反馈

------

【八、API 契约（必须使用的具体接口）】

所有页面必须基于以下已有接口生成原型或实现，不允许虚构新的后端接口。

通用认证约束：

- 所有任务业务接口和图表分析接口均需要携带登录态：
  - `Authorization: Bearer <token>`
- 统一响应结构遵循：
  - 成功：`{ code: 200, message: string, data: ... }`
  - 分页：`{ code: 200, message: string, data: { records, total, page, pageSize } }`

页面A：任务工作台首页

- 重点任务卡片：
  - `POST /data_management/tasks/find_task_params`
    - 用途：获取左侧重点任务卡片
    - 请求参数使用已有字段：`page`、`page_size`、`inspection_id`、`repairman_id`、`area_id`、`pipe_id`、`type`、`priority`、`status`、`start_time`、`end_time`
    - 与页面B表格使用同一个接口
    - 页面A只取少量重点任务，不取全量后堆卡片
- 任务统计摘要：
  - `GET /data_management/tasks/Indicator_card`
    - 用途：获取任务统计指标，如总数、待处理、进行中、已完成、已取消
- 任务详情模态框：
  - `POST /data_management/tasks/find_task_id`
    - 用途：点击重点任务卡片后获取任务详情并弹出模态框
    - 请求体使用任务 `id`
- 任务状态分布饼图：
  - `POST /data_visualization/task_details/task_status`
    - 参数：`area_id`、`start_time`、`end_time`
- 任务数量趋势折线图：
  - `POST /data_visualization/task_details/task_nums`
    - 参数：`area_id`、`start_time`、`end_time`
- 区域任务对比柱状图：
  - `POST /data_visualization/task_details/task_contrast`
    - 参数：`area_id`、`start_time`、`end_time`
- KPI榜单：
  - `POST /data_visualization/task_details/KPI_list`
    - 参数：`area_id`、`type`、`start_time`、`end_time`
    - `type` 含义：`0` 任务完成数，`1` 响应时间，`2` 满意度

页面B：所有任务列表页

- 高密度二维表：
  - `POST /data_management/tasks/find_task_params`
    - 用途：页面B主表格查询
    - 必须和页面A重点任务卡片使用同一个列表接口
    - 页面B可使用完整分页和筛选参数
- 任务详情模态框：
  - `POST /data_management/tasks/find_task_id`
    - 用途：点击表格行或详情按钮后获取任务详情并弹出模态框
- 可选操作接口：
  - `POST /data_management/tasks/add_task`
  - `POST /data_management/tasks/update_task`
  - `GET /data_management/tasks/delete_task/{id}`
  - 如果当前原型只做浏览和详情，不要强行展示新增、修改、删除流程

接口与页面映射限制：

- 页面A重点任务卡片和页面B高密度表格必须使用同一个接口：`POST /data_management/tasks/find_task_params`
- 页面A图表只能使用 `docs/api/02-数据可视化模块.md` 模块2的图表接口
- 模态框详情必须使用 `POST /data_management/tasks/find_task_id`
- 不允许新增“任务态势统一查询接口”“重点任务专用接口”“图表聚合接口”等后端不存在的接口
- 不允许用前端假数据替代已有接口字段，除非明确标注为静态原型占位

【九、扩展能力（Advanced Features，可选）】

- 任务优先级看板视图
- 时间顺序任务流视图
- 任务切片快速筛选
- KPI榜单类型切换：任务完成数 / 响应时间 / 满意度
- 页面B保存筛选视图

【十、设计目标（Design Goals）】

- 页面A不是纯卡片式布局，必须包含饼图、折线图、柱状图和 KPI 榜单
- 页面A左侧只展示少量重点任务，并提供“所有任务”跳转
- 页面A右上角筛选框必须可展开、可压缩，且不挤压主布局
- 页面B使用高密度二维表承接所有任务
- 页面A卡片和页面B表格使用同一个任务列表接口
- 任务详情使用模态框，不使用常驻抽屉，不新增独立详情页
- 保持冷硬监控台气质，不做营销式数据大屏

------

【十一、限制与规范（Constraints）】

- 业务来源：原项目“任务管理”模块
- 图表来源：`docs/api/02-数据可视化模块.md` 中的模块2“任务详情分析”
- 页面A左侧重点任务不允许无限堆卡片
- 页面A右侧必须包含三个图表和 KPI 榜单
- 页面B必须使用高密度二维表
- 具体接口必须遵守本提示词“API 契约”章节
- 所有功能必须基于现有任务管理与数据可视化接口
- 不使用圆角（borderRadius: 0px）
- 不使用纯灰色（#808080）
- 不为了交互完整而凭空增加后端没有的操作流
