# 应急模块生成提示词

设计一个【应急演练管理页面】，用于【基于现有演练管理接口查询、筛选、查看、新增、修改和删除演练信息】。

本提示词的核心约束是：页面必须 API 驱动，只能围绕 `docs/api/09-演练管理模块.md` 已有接口设计功能。不要虚构事故响应、实时指挥、资源调度、处置闭环、阶段节点写入等后端未提供的功能。

==================================================

【一、页面定位（Page Role）】

- 页面类型：演练管理工作台
- 用户角色：演练管理人员、应急指挥人员、调度值班人员
- 核心目标：
  - 目标1：查询和筛选演练信息
  - 目标2：以时间流方式浏览演练开始、结束、状态和历史记录
  - 目标3：查看单个演练详情，包括结果和问题记录
  - 目标4：在接口支持范围内新增、修改、删除演练信息
- 使用场景：
  - 按名称、类型、状态、区域、时间范围筛选演练
  - 查看某次演练的基础信息、参与人数、组织部门、结果、问题
  - 维护演练记录，不扩展成事故响应系统

------

【二、整体布局（Layout System）】

- 顶部结构：固定主导航 + 页面标题 + 主要操作按钮
- 主体布局：筛选区 + 时间流列表 + 右侧详情/摘要区
- 区域划分：
  - 顶部筛选区：名称、类型、状态、区域、时间范围、查询、重置
  - 左侧/主区域：演练时间流列表
  - 右侧辅助区：当前选中演练摘要、统计摘要、操作入口
  - 弹窗/抽屉：新增演练、编辑演练、查看详情、删除确认
- 是否使用特殊布局：
  - 可以使用 Timeline 展示演练记录
  - 不做复杂阶段编排
  - 不做实时指挥大屏

------

【三、信息架构（Information Architecture）】

模块1：筛选工作区

- 功能说明：基于接口已有参数查询演练信息
- 包含信息：
  - 演练名称 `name`
  - 演练类型 `type`
  - 演练状态 `status`
  - 区域 `area_id`
  - 开始时间 `start_time`
  - 结束时间 `end_time`
- 数据来源：
  - `POST /manoeuvre/find_manoeuvre_params`
- 优先级：最高

模块2：演练时间流列表

- 功能说明：以时间顺序展示演练记录
- 包含信息：
  - 演练ID
  - 演练名称
  - 演练类型
  - 演练状态
  - 区域名称
  - 演练地点
  - 开始时间
  - 结束时间
  - 参与人数
  - 组织部门
  - 描述
- 数据来源：
  - `POST /manoeuvre/find_manoeuvre_params`
- 展示规则：
  - 时间流只是列表展示方式
  - 每一条时间流记录对应一个演练对象
  - 不拆出后端不存在的“阶段节点”或“处置步骤”
- 优先级：最高

模块3：演练详情视图

- 功能说明：展示单个演练详情
- 包含信息：
  - 列表字段全部信息
  - 演练结果 `result`
  - 发现问题 `issues`
- 数据来源：
  - `GET /manoeuvre/find_manoeuvre_id`
- 触发方式：
  - 点击时间流记录
  - 点击“查看详情”按钮
- 展示方式：
  - 右侧详情面板或模态框均可
  - 不新增独立详情页
- 优先级：高

模块4：新增/编辑演练表单

- 功能说明：在接口支持范围内维护演练信息
- 新增数据来源：
  - `POST /manoeuvre/add_manoeuvre`
- 编辑数据来源：
  - `POST /manoeuvre/update_manoeuvre`
- 表单字段必须来自接口文档，不新增后端未支持字段：
  - `name`
  - `type`
  - `area_id`
  - `location`
  - `start_time`
  - `end_time`
  - `participants`
  - `organizer`
  - `description`
  - 编辑时可包含 `id`、`status`、`result`、`issues`
- 优先级：中高

模块5：删除确认

- 功能说明：删除演练信息前进行确认
- 数据来源：
  - `GET /manoeuvre/remove_manoeuvre`
- 交互规则：
  - 删除前必须弹出确认
  - 删除成功后刷新列表
  - 不做批量删除，除非接口明确支持
- 优先级：中

------

【四、核心模块详细设计（Key Module Design）】

模块名称：筛选工作区

- 展示形式：紧凑表单栏
- 控件：
  - 演练名称输入框
  - 类型选择
  - 状态选择
  - 区域选择
  - 时间范围选择
  - 查询按钮
  - 重置按钮
- 行为：
  - 点击查询调用 `POST /manoeuvre/find_manoeuvre_params`
  - 重置后清空筛选条件并重新查询
  - 不新增接口未支持的筛选条件

模块名称：演练时间流列表

- 展示形式：纵向 Timeline
- 每条记录包含：
  - 时间范围：`start_time` 至 `end_time`
  - 演练名称：`name`
  - 类型：`type`
  - 状态：`status`
  - 区域与地点：`area_name`、`location`
  - 参与人数：`participants`
  - 组织部门：`organizer`
  - 简要描述：`description`
- 操作：
  - 查看详情
  - 编辑
  - 删除
- 限制：
  - 不把一条演练拆成多个“阶段节点”
  - 不生成后端没有的“处置节点”“应急资源”“指挥链路”

模块名称：演练详情面板/模态框

- 数据来源：`GET /manoeuvre/find_manoeuvre_id`
- 请求参数：
  - `id`
- 展示内容：
  - 演练基础信息
  - 时间范围
  - 参与人数
  - 组织部门
  - 描述
  - 结果 `result`
  - 问题 `issues`
- 行为：
  - 打开详情时按 ID 查询最新数据
  - 关闭后保留当前筛选和列表状态

模块名称：新增/编辑演练弹窗

- 新增接口：`POST /manoeuvre/add_manoeuvre`
- 编辑接口：`POST /manoeuvre/update_manoeuvre`
- 表单规则：
  - 新增时填写名称、类型、区域、地点、开始时间、结束时间、参与人数、组织部门、描述
  - 编辑时允许修改状态、结果、问题等接口支持字段
  - 表单字段不得超过 API 文档范围
- 提交后：
  - 成功则关闭弹窗并刷新列表
  - 失败则展示错误信息

------

【五、交互模型（Interaction Model）】

- 主交互方式：
  - 筛选查询
  - 时间流浏览
  - 点击记录查看详情
  - 新增/编辑/删除演练
- 分层交互：
  - 第一层：筛选条件控制演练列表
  - 第二层：时间流记录展示演练概况
  - 第三层：详情面板/模态框展示单条演练完整信息
  - 第四层：新增/编辑弹窗维护演练记录
- 页面跳转策略：
  - 不新增详情页
  - 不跳转到事故响应页面
  - 所有操作在当前页面内完成
- 动态行为：
  - 查询时列表刷新
  - 点击详情时打开详情层
  - 新增/编辑/删除成功后刷新列表

------

【六、视觉与层级（Visual Hierarchy）】

- 视觉优先级：
  1. 当前筛选结果中的演练时间流
  2. 当前选中演练详情
  3. 筛选与维护操作
  4. 右侧摘要信息
- 设计风格：Tactical Brutalism — 冷硬监控台语言，强调记录、时间、状态和可追溯性
- 组件风格：
  - 时间流记录：0px 圆角，清晰边界，紧凑信息密度
  - 筛选栏：高密度表单，不做大面积装饰
  - 详情层：信息分组清楚，结果和问题字段突出但不夸张
  - 操作按钮：只展示接口支持的新增、编辑、删除、查看详情
- 状态颜色规范：
  - 未开始：弱化状态
  - 进行中：橙色强调
  - 已完成：青色或绿色
  - 异常/问题：黄色或橙色提示

------

【七、动效设计（Motion & Feedback）】

- Hover 动效：时间流记录背景轻微变化
- 查询反馈：列表加载时使用克制进度条或扫描线
- 弹窗动效：新增/编辑/详情弹窗短促出现，不做复杂转场
- 删除反馈：删除前确认，删除后 toast 或状态提示

------

【八、API 契约（必须使用的具体接口）】

所有页面功能必须基于以下已有接口，不允许虚构新接口。

通用认证约束：

- 所有演练管理接口均需要携带登录态：
  - `Authorization: Bearer <token>`
- 统一响应结构遵循：
  - 成功：`{ code: 200, message: string, data: ... }`

查询演练列表：

- `POST /manoeuvre/find_manoeuvre_params`
- 用途：
  - 页面初始化加载演练列表
  - 筛选后刷新演练时间流
- 请求参数：
  - `name`
  - `type`
  - `status`
  - `area_id`
  - `start_time`
  - `end_time`
- 响应字段用于列表展示：
  - `id`
  - `name`
  - `type`
  - `status`
  - `area_name`
  - `location`
  - `start_time`
  - `end_time`
  - `participants`
  - `organizer`
  - `description`

查询演练详情：

- `GET /manoeuvre/find_manoeuvre_id`
- 用途：
  - 点击时间流记录或“查看详情”后获取单条演练详情
- 请求参数：
  - `id`
- 响应字段用于详情展示：
  - 列表字段
  - `result`
  - `issues`

新增演练：

- `POST /manoeuvre/add_manoeuvre`
- 用途：
  - 新增演练信息
- 请求字段：
  - `name`
  - `type`
  - `area_id`
  - `location`
  - `start_time`
  - `end_time`
  - `participants`
  - `organizer`
  - `description`

修改演练：

- `POST /manoeuvre/update_manoeuvre`
- 用途：
  - 修改演练信息、状态、结果或问题
- 请求字段：
  - `id`
  - `status`
  - `result`
  - `issues`
  - 以及接口文档允许的其他演练字段

删除演练：

- `GET /manoeuvre/remove_manoeuvre`
- 用途：
  - 删除单条演练信息
- 请求参数：
  - `id`

接口与功能限制：

- 不允许新增事故响应接口
- 不允许新增演练阶段节点接口
- 不允许新增实时指挥、资源调度、人员签到、任务派发等接口
- 不允许把后续扩展方向写成当前页面功能
- 如果原型需要展示“阶段感”，只能由 `start_time`、`end_time`、`status`、`result`、`issues` 等已有字段进行展示层编排

------

【九、扩展能力（仅可标记为后续方向，不得作为当前功能）】

- 演练阶段节点管理
- 演练参与人员签到
- 应急资源调度
- 事故响应闭环
- 实时指挥态势

以上能力当前接口不支持，只能作为后续扩展方向，不得直接生成当前页面功能。

------

【十、设计目标（Design Goals）】

- 严格 API 驱动，所有当前功能都有 `docs/api/09-演练管理模块.md` 接口支撑
- 保留时间流体验，但时间流只展示演练记录，不虚构阶段节点
- 提供高效筛选、查看详情、新增、编辑、删除能力
- 保持冷硬监控台气质，不做营销式应急大屏

------

【十一、限制与规范（Constraints）】

- 业务来源：原项目“模拟与演练”模块
- 移除事故响应模块，不生成事故响应相关功能
- 当前阶段只基于现有演练管理接口能力来做
- 必须明确区分“当前可实现功能”和“后续扩展方向”
- 不使用圆角（borderRadius: 0px）
- 不使用纯灰色（#808080）
- 不为了交互完整而凭空增加后端没有的操作流
