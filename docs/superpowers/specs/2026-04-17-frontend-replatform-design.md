# 前端重构与 Vue 3 迁移设计草案

## 1. 背景与目标

本次前端重构有两个明确目标：

- 将后台前端从 Vue 2 迁移到 Vue 3。
- 在迁移过程中重构 UI 设计，但不引入“AI 味”模板化后台。

当前仓库处于双轨状态：

- `backstage-vue` 是现有 Vue 2 + Vue CLI + Element UI 后台，承载已有业务。
- `frontend` 是新的 Vue 3 + Vite 工程，目前仍是空壳，尚未承接真实业务。

本轮讨论已经明确，不采用在 `backstage-vue` 上原地升级的方式，而是采用 **Vue 3 新工程承接旧业务、逐步迁移** 的路径。

## 2. 已确认决策

### 2.1 总体策略

- 本次采用“技术迁移优先，UI 分阶段升级”的策略。
- 第一阶段优先目标不是全量业务迁移，而是先建立 Vue 3 技术底座和新的设计系统。
- UI 重构先做设计系统与框架样板，待验收通过后再进入页面迁移。

### 2.2 第一阶段迁移范围

第一阶段仅覆盖以下内容：

- 登录页
- 后台主框架
- Dashboard 首页

选择这一范围的原因：

- 登录页可以先跑通认证链路、表单、请求层、路由守卫。
- 后台主框架可以先稳定全局布局、模块切换、状态摘要与导航结构。
- Dashboard 可以作为视觉样板页，同时验证图表、地图、KPI、告警流等高价值模块。

### 2.3 组件体系策略

- 不继续依赖现成后台组件库塑造整体视觉。
- 采用自建 UI 组件体系。
- 设计系统必须先行，并作为单独验收物。
- 设计系统通过验收后，才继续批量迁移业务页面。

### 2.4 后端约束

- 不为这次前端重构额外新增后端业务。
- 新框架中的左侧工作区、筛选区、上下文区，优先复用当前已有业务维度与已有数据结构。

## 3. 设计方向总纲

### 3.1 核心定位

本系统不做泛化 SaaS 后台，也不做概念化赛博界面。整体设计基调定义为：

- 冷硬监控台
- 少量高端编辑感

其中：

- “冷硬监控台”负责建立秩序、精度、压迫感、决策效率和工业控制感。
- “高端编辑感”只用于提升信息层级、版式节奏、留白纪律和文本质感。

因此，这套系统的视觉主语必须始终是 **控制室 / 调度台 / 仪表面板**，而不是杂志站点、营销页或 Dribbble 式概念稿。

### 3.2 设计原则

- 不允许回到“左侧导航栏 + 右侧内容区”的传统后台骨架。
- 不允许大量圆角、柔软投影、紫白渐变、过度通用字体等常见 AI 后台特征。
- 页面层级优先通过表面明暗差建立，而不是依赖粗边框或泛滥阴影。
- 强调色必须节制使用，只服务关键状态、操作或风险提示。
- 风格集中在入口页和核心总览页，工作流页面必须收敛，优先保证效率。

## 4. 页面分级规则

为避免“所有页面都像视觉海报”，页面按风格承载能力分为三类：

### A 级页面：品牌展示页

适用页面：

- 登录页
- Dashboard 首页

特点：

- 承载最强的系统识别度
- 允许更明显的明暗对比、排版张力、品牌语气与氛围纹理
- 允许更强的强调色使用，但仍需服务信息层级

### B 级页面：分析操作页

适用页面：

- 区域详情
- 管线详情
- 监控分析页

特点：

- 保留主要视觉语言，但明显降低装饰密度
- 优先突出数据阅读、筛选和图表结构
- 动效、发光和纹理都必须明显收敛

### C 级页面：工作流页面

适用页面：

- 表格页
- 表单页
- 配置页
- 抽屉、弹窗等操作密集页

特点：

- 保留同一套设计系统的材料、字体和状态语言
- 风格表达降到 20% 至 30%
- 可读、可扫、可填、可批量操作优先于风格展示

## 5. 颜色与表面层级

### 5.1 整体色彩方向

不使用“纯黑 + 霓虹”的陈旧科幻路线，也不使用泛深色后台模板。整体推荐方向为：

- 深石墨
- 冷蓝灰
- 矿物感深底

目标是在视觉上更接近冷金属、石墨面板和控制室材料，而非夜店科技风。

### 5.2 表面角色

页面中的颜色按功能角色划分：

- `Base / Background`
  系统最底层背景，负责压住整体情绪，不使用纯黑。
- `Panel`
  一级结构区，作为主框架和主功能区的承载层。
- `Module`
  模块、卡片、图表容器等主要工作区域。
- `Overlay`
  浮层、弹窗、抽屉等临时高关注层。
- `Text`
  标题、正文、次级信息、数据标签的文字层。
- `Accent`
  主强调色，偏安全橙，用于关键动作、选中态和高价值指标。
- `Signal`
  辅助强调色，偏冷青，用于系统提示、focus 和智能反馈。
- `Warning / Danger / Success`
  用于告警、风险、正常状态，但整体需克制，接近工业信号系统。

### 5.3 表面逻辑约束

- 大部分结构层级通过表面亮度变化建立，不依赖大面积边框。
- 强调色不允许大面积铺设为氛围背景，只能点亮关键节点。
- Overlay 可使用轻度模糊与更强分离度，但不能发展成浮夸玻璃风。

## 6. 布局策略

### 6.1 已排除方案

传统后台常见的以下布局已被明确排除：

- 左侧导航栏 + 右侧内容区

原因：

- 即使更换颜色和组件，也容易回到“普通后台壳”。
- 无法体现控制台、调度台、工作台的系统气质。

### 6.2 已选定主框架

主框架采用：

- 顶部主导航 + 左右双区工作台

这不是网站式布局，而是调度工作台布局。

### 6.3 框架三分区

#### 顶部主导航

职责：

- 一级模块切换
- 全局状态与系统摘要
- 系统名、告警概览、全局时钟等总控信息

不承担：

- 页面内局部筛选
- 业务细粒度操作

#### 左侧工作台

职责：

- 当前视角控制
- 运行上下文提示
- 风险切片与状态筛选
- 当前关注事项摘要

它不是导航栏，而是“当前看什么、看哪一片、看哪种状态”的控制器。

#### 右侧主工作区

职责：

- 承载页面主体内容
- 展示地图、图表、表格、列表、详情和工作流内容

### 6.4 各页面的框架适配

- 登录页：不使用三分区工作台，但沿用同一材料、颜色和字体系。
- Dashboard：完整使用三分区框架，是整套系统的标准样板页。
- 分析类页面：继续使用完整框架，但左侧工作台内容按页面收敛。
- 表格/表单类页面：仍保留顶部与左侧，但左侧会变轻，右侧让位于工作流内容。

## 7. 左侧工作台信息架构

左侧工作台优先使用现有业务维度，不新增后端业务。

### 7.1 可直接复用的现有业务维度

基于当前 Vue 2 工程中的路由、Dashboard、监控、管理等模块，以下内容已具备复用条件：

- 区域维度
- 管线维度
- 运行状态维度
- 时间范围维度
- 告警与任务语义

### 7.2 左侧工作台的优先结构

第一阶段建议左侧工作台固定包含以下 4 组内容：

- 区域上下文
- 管线或对象范围
- 状态筛选
- 时间窗

可选补充：

- 当前关注事项摘要
  包括高风险告警、逾期任务、关键设备等摘要信息

### 7.3 左侧工作台定位

左侧工作台明确偏向：

- 态势摘要 + 少量筛选

不建议演变为纯筛选器面板，也不建议变成目录索引式导航。

### 7.4 左侧工作台形态

左侧工作台已进一步确认采用：

- 摘要块

这意味着：

- 左侧优先展示当前上下文和态势摘要，而不是完整筛选器集合。
- 交互筛选可以存在，但必须作为轻量附着能力出现。
- 用户首先读取“当前局面”，随后才决定是否切换视角或下钻。

### 7.5 左侧工作台标准顺序

第一阶段左侧工作台建议固定为以下四段：

1. `当前视角`
   包括区域、管线、时间窗等上下文信息。
2. `运行态势`
   包括正常、异常、离线、高风险等状态摘要。
3. `关注事项`
   包括高风险告警、逾期任务、待处置项等关键提醒。
4. `快速切换`
   提供轻量级区域、管线、时间视角切换入口。

该顺序的目标是保证左侧工作台的阅读逻辑清晰：

- 先明确当前看什么
- 再理解当前稳不稳
- 再识别最该关注什么
- 最后决定是否切换视角

## 8. 设计系统验收结构

这套设计系统后续将被整理为可验收的独立 spec。当前已确认后续验收文档需覆盖以下模块：

- 设计总纲
- Brand DNA
- Typography
- Layout
- Component
- Motion
- Page Grading

其中当前已讨论并形成共识的部分主要集中在：

- 设计总纲
- 页面分级
- 颜色与表面层级
- 主框架定义
- 左侧工作台定位

## 9. Typography 与信息密度

### 9.1 中文优先原则

项目中的主要阅读对象为中文界面文本，因此字体系明确采用：

- 中文优先

这意味着：

- 设计个性不依赖英文展示字体建立。
- 视觉识别主要来自版式秩序、字重、字距、数字系统和模块节奏。
- 中文标题和正文必须首先保证稳定、耐读和长时间使用不疲劳。

### 9.2 字体系角色分工

后续字体系统按三类角色定义：

- `Display / Section Title`
  用于登录页标题、Dashboard 主区标题、一级模块名。
- `Body / UI Text`
  用于正文、按钮、表头、筛选项、表单和说明信息。
- `Data / Numeric`
  用于 KPI、监测值、时间戳、设备编号、状态计数和趋势读数。

### 9.3 Typography 规则

- 标题字与正文必须分家，不采用“一套字体覆盖全部角色”的方式。
- 正文字体系优先服务高频操作和信息扫描，不追求夸张个性。
- 标题表达可以更硬朗，但不允许为了风格牺牲汉字识别效率。
- 数据字必须单独强化，保证数字跳动稳定、对齐清晰、扫描迅速。

### 9.4 信息密度

当前信息密度已定为：

- 中高密度

对应约束如下：

- Dashboard 保持较强信息量，但模块之间仍需具备明确呼吸感。
- 卡片、趋势、告警、列表可以同屏共存，不做松散大留白布局。
- 表格、列表、筛选区比营销型页面更紧凑，但不走极限监控屏压缩路线。
- 编辑感来自排版秩序和节奏，而非大面积留白。

### 9.5 单屏约束

Dashboard 首页进一步增加如下硬约束：

- 第一屏核心内容必须单屏呈现完成
- 默认不依赖页面级纵向滚动条

这意味着：

- 首页模块数量必须受到严格控制
- 每个区块优先展示摘要和关键态势，而不是展开完整分析
- 首页是指挥总览页，不是深度分析页

## 10. Dashboard 首页结构

### 10.1 首页中心策略

Dashboard 首页已确认采用：

- 地图中心

地图是首页唯一绝对主角，用于承接空间关系、区域热度、点击钻取与态势总览。

### 10.2 首页骨架

Dashboard 第一屏按以下结构组织：

- `中心主画面`
  中国区域监控地图或主区域地图。
- `上缘信息带`
  压缩型全局 KPI 信息条，用于展示总量、异常数、活跃任务、在线设备等核心总览。
- `左侧工作台`
  展示区域、管线、状态、时间四组上下文与筛选。
- `右侧态势区`
  展示风险态势与处置态势。

### 10.2.1 主工作区视觉组织

右侧主工作区已确认采用：

- 强模块拼接感

这意味着：

- 主工作区不是一张连续大画布，而是明确分区的调度台面板。
- 每个模块都应具备清晰边界、稳定职责和可独立识别的结构轮廓。
- 模块之间通过硬朗拼接关系建立秩序，而不是依赖柔和留白弱连接。
- 地图虽然是首页主角，但不能吞没其他信息区块。

对应的视觉约束如下：

- 模块边界清晰，但不使用夸张粗边框。
- 模块拼接优先依赖表面层级、间距纪律和结构线索。
- 各模块必须像“被装配到同一张控制台上”，而不是像一组自由漂浮卡片。

### 10.3 右侧态势区优先级

右侧态势区的优先级已确认如下：

1. 告警与风险优先
2. 任务与处置进度次优先

因此右侧态势区建议拆分为上下两段：

- `上段：风险态势`
  包括高风险告警数、最新告警流、告警等级分布、重点异常区域或管线。
- `下段：处置态势`
  包括处理中任务、逾期任务、待派发任务、最新处置进展。

### 10.4 下缘趋势区策略

原“下缘趋势区”已从 Dashboard 首页第一屏中移除。

原因如下：

- 首页被要求在单屏内完整显示核心内容
- 地图中心、右侧风险 / 处置和左侧摘要块已构成完整指挥视图
- 趋势分析应下沉到二级分析页或详情页，不再占用首页首屏空间

### 10.5 首页模块关系

Dashboard 首页各模块的关系已收敛为：

- 地图模块：绝对主模块
- 右侧风险 / 处置模块：高优先侧模块
- 上缘 KPI 信息带：背景支撑模块

阅读顺序明确为：

1. 先看地图态势
2. 再看右侧风险与处置
3. 最后通过 KPI 信息带补全总览认知

### 10.6 单屏首页约束

Dashboard 首页作为总览工作台，必须满足：

- 关键内容在常规桌面视口内无需纵向滚动即可读完
- 模块高度和信息密度服从单屏约束
- 深度趋势、长列表、扩展明细应转移到二级页面或弹出层

## 11. 登录页方向

登录页属于 A 级高风格承载页。

当前已确认其气质定位为：

- 更像控制中心前厅
- 有展示感，但仍偏冷硬

这意味着：

- 登录页需要具备系统入口仪式感，但不能做成营销型欢迎页。
- 允许更强的标题张力、材料感和空间感。
- 仍需保持工业控制语气，与 Dashboard 首页视觉连续。

### 11.1 登录页结构

登录页结构已确认采用：

- 左侧品牌叙事区 + 右侧登录卡

其中：

- 左侧品牌叙事区负责建立系统气质、责任感和世界观，不承担营销展示职责。
- 右侧登录卡负责纯操作入口，承担登录表单、状态提示和错误反馈。

### 11.2 左侧品牌叙事区定位

左侧叙事区当前确定以：

- 系统使命感

为主，而不是以实时态势展示为主。

表达重点包括：

- 守护能源基础设施安全运行
- 强调长期稳定、风险预警、调度保障和处置能力
- 建立系统级责任感，而非单纯展示技术炫技

建议内容结构收敛为三层：

1. 系统名称
2. 使命型主文案
3. 2 至 3 条支撑性能力摘要

### 11.3 登录页文案语气

登录页文案语气已确认采用：

- 正式里带一点技术锋利感

约束如下：

- 不使用广告式夸张话术。
- 句子保持短促、克制、密度高。
- 关键词围绕管网安全、实时监测、风险预警、调度处置、持续运行。
- 允许少量技术锋利感，但不能滑向概念化宣传语。

## 12. 顶部主导航

顶部导航当前已确认采用：

- 一级模块 + 少量全局状态块

### 12.1 顶部导航职责

顶部导航负责：

- 一级模块切换
- 系统名称或当前工作域识别
- 全局告警摘要
- 系统状态或时间信息

顶部导航不负责：

- 页面内局部筛选
- 复杂业务操作
- 承担二级导航堆叠

### 12.2 顶部导航设计约束

- 顶部导航应表现为总控台上沿，而不是普通网站 header。
- 视觉上必须紧凑、稳定、克制，不做悬浮营销头图。
- 全局状态块数量必须少，避免挤占一级模块切换的识别效率。
- 一级模块优先使用短名称，减少导航拥堵。

## 13. 字体参考修正

用户提供了 Anthropic / Claude 的字体参考：

- 官网标题、副标题使用 `Styrene`
- 官网正文使用 `Tiempos`
- Claude 对话主体以 `Styrene B` 为主
- UI 部分使用定制字体 `Anthropic Sans`

这一参考的价值不在于照搬字体本身，而在于其组合逻辑：

- 无衬线字体负责界面秩序、亲和感与现代感
- 衬线字体负责提升阅读质感与编辑感
- 字重和字形本身具备足够个性，因此不依赖夸张装饰

### 13.1 对当前项目的适用性判断

该参考不能直接原样套用，原因如下：

- 当前项目几乎全部为中文界面，英文品牌字体逻辑不能直接迁移
- `Styrene`、`Styrene B`、`Tiempos` 均为商业字体，不适合作为默认可落地依赖
- 当前系统目标不是 Anthropic 式“温和亲切”，而是“冷硬监控台 + 少量编辑感”

### 13.2 可借鉴原则

该参考中可直接借鉴的不是具体字体名，而是以下原则：

- 中文界面字体应选择稳定、克制、耐读的无衬线主字体
- 识别度应来自字重、字距、数字系统和版式节奏，而不是花哨标题字
- 如需引入编辑感，应极轻量地用于标题或局部说明，而不是侵入整个 UI
- 数据字系统仍需独立强化，保证监控场景下的数值稳定性和扫描效率

### 13.3 当前字体方向更新

结合“中文优先”和 Anthropic 参考，当前字体方向更新为：

- 主 UI 字体：中文优先、稳定耐读、克制的无衬线体系
- 标题风格：不追求夸张展示感，而通过字重、字距和编排获得质感
- 数据字体：独立的数据字体系，承担控制台气质
- 编辑感来源：局部排版质感，而非大面积 serif 介入

### 13.4 已选定字体策略

当前已正式选定：

- 方案 B：监控台 + 一点编辑感

建议的字体结构如下：

- UI / 正文主字体：`Noto Sans SC` 或 `Source Han Sans SC`
- 局部标题 / 局部强调：`Noto Serif SC` 或 `Source Han Serif SC`
- 数据字：`JetBrains Mono` 或 `IBM Plex Mono`

### 13.5 字体使用边界

为避免字体系统失控，使用边界固定如下：

- 无衬线字体承担绝大多数 UI、正文、表单、导航、标签和表头场景。
- 衬线字体只允许小范围用于登录页主文案、Dashboard 局部标题或少量说明性标题。
- 数据字体只用于 KPI、监测值、时间戳、设备编号、状态计数等数值密集场景。
- 不允许在高频操作区、表格主体、长列表中大面积使用 serif。

### 13.6 视觉目标

该方案的目标不是制造“文学感”，而是：

- 以无衬线维持系统秩序和操作效率
- 用极少量 serif 注入编辑感和质感
- 通过数据字强化控制台识别度
- 保持整体冷硬，不走温和品牌站路线

### 13.7 字号层级

结合“中高密度”与“单屏总览”要求，字体层级建议按以下区间控制：

- `Display`
  32px 至 44px
  用于登录页主标题、Dashboard 核心区块标题中的极少数场景。
- `Headline`
  22px 至 28px
  用于页面主标题、关键模块标题。
- `Section Title`
  16px 至 20px
  用于模块区块标题、地图侧模块标题、重点面板标题。
- `Body`
  13px 至 15px
  用于正文、说明、列表正文、表单说明和常规信息文本。
- `Label / Meta`
  11px 至 12px
  用于表头、辅助标签、状态说明、时间说明、次级注释。
- `Data`
  18px 至 32px
  按重要程度用于 KPI、计数值、风险数字和监测读数。

### 13.8 字重规则

字重按功能约束如下：

- `Regular`
  用于正文、说明、辅助信息。
- `Medium`
  用于按钮、导航、模块标题、小标签、表头。
- `Semibold / Bold`
  只用于主标题、关键数据和极少数重点强调。

约束如下：

- 不依赖极重字重制造压迫感，压迫感主要由布局、对齐和模块关系提供。
- 业务页内避免大量 Bold 并存，以免破坏扫描秩序。

### 13.9 字距规则

字距用于建立“控制台纪律感”，但必须克制：

- 登录页主标题、Dashboard 一级标题允许轻微放大字距，强调系统性和秩序感。
- 模块标题和导航项可以有轻度 tracking，形成硬朗感。
- 正文、表格内容、表单输入不主动拉大字距，避免影响中文阅读效率。
- Label 层允许更明显的小范围字距扩展，用于形成设备标牌感和元信息层级。

### 13.10 行高规则

行高策略按密度控制如下：

- Display / Headline：1.1 至 1.2
- Section Title：1.2 至 1.35
- Body：1.5 左右
- Label / Meta：1.3 至 1.4
- Data：1.0 至 1.15

约束如下：

- 首页与工作台场景中，正文行高不能过大，否则会破坏单屏密度。
- 登录页左侧叙事区行高可略放松，但仍需保持克制。
- 数据字行高必须紧凑，避免 KPI 卡片显得松散。

### 13.11 Typography 应用建议

基于当前系统目标，建议采用以下排版分工：

- 登录页系统名：无衬线，较高字重，允许较强字距控制
- 登录页使命型主文案：可小范围尝试 serif，承担质感提升
- Dashboard 模块标题：以无衬线为主，保持稳定和硬朗
- 状态标签与导航：无衬线 `Medium`
- KPI 与时间戳：数据字系统
- 表格、筛选、表单：无衬线正文体系，不引入 serif
## 14. 第一批组件优先级

为保证第一阶段能够先验收系统气质与框架质量，而不是陷入零散基础组件细节，第一批组件按以下优先级推进。

### 13.1 框架级组件

最高优先级组件如下：

- 顶部主导航
- 左侧摘要块工作台
- 主工作区容器
- 模块面板壳

这些组件决定整套系统最核心的视觉骨架和布局秩序，是 Vue 3 新工程的第一批必须落地项。

### 13.2 信息展示组件

第二优先级组件如下：

- KPI 信息条
- 状态摘要块
- 告警流
- 任务摘要块
- 趋势图容器

这些组件决定 Dashboard 首页是否能具备真实的监控台阅读逻辑。

### 13.3 交互基础组件

第三优先级组件如下：

- 按钮
- 输入框
- 轻量切换器
- 状态标签

这批组件用于支撑登录页、顶部导航状态块、左侧工作台快速切换等轻交互场景。

### 13.4 业务承载组件

第四优先级组件如下：

- 表格
- 筛选条
- 详情模块
- 图表组合块

这批组件在第一阶段不抢先做完，而是在样板页气质已经成立后，再按页面迁移需要补足。

## 15. 样板页开发顺序

第一阶段的样板页开发顺序建议固定如下：

### 14.1 第一步：主框架样板

先落地以下内容：

- 顶部主导航
- 左侧摘要块工作台
- 右侧主工作区骨架
- 模块拼接关系

目标是先验证整套系统的空间组织和信息架构是否成立。

### 14.2 第二步：Dashboard 首页样板

在主框架稳定后，再落地：

- 地图中心首页
- 右侧风险 / 处置态势区
- 下缘主趋势区
- 上缘 KPI 信息带

目标是验证整套系统最强视觉承载页是否达到预期。

### 14.3 第三步：登录页样板

在 Dashboard 样板成立后，再落地：

- 左侧品牌叙事区
- 右侧登录卡
- 中文文案气质
- 登录状态与错误反馈

目标是验证系统入口页能否同时具备使命感、控制感和可操作性。

### 14.4 第四步：基础组件固化

在样板页确认后，再反向抽取和固化：

- 颜色 token
- 字号与密度规则
- 面板、按钮、标签、输入框等组件规范

目标是避免一开始脱离页面上下文做组件，导致组件系统与真实页面气质脱节。

## 18. 当前结论摘要

截至 2026-04-17，本轮讨论已形成以下稳定结论：

- Vue 3 迁移采用新工程承接旧业务的方式推进。
- 第一阶段先做登录页、后台主框架和 Dashboard。
- UI 先做设计系统并先验收，再进入页面批量迁移。
- 设计方向是“冷硬监控台 + 一点高端编辑感”。
- 主框架采用“顶部主导航 + 左右双区工作台”。
- 左侧不做导航，而做“态势摘要 + 少量筛选”的工作台。
- 左侧优先复用区域、管线、状态、时间四类现有业务维度。
- Typography 采用中文优先策略，信息密度定为中高密度。
- Dashboard 首页采用地图中心结构。
- 右侧态势区优先展示告警与风险，其次展示任务与处置进度。
- Dashboard 首页按单屏无纵向滚动约束组织，不再保留底部趋势图区。
- 右侧主工作区采用“强模块拼接感”，以分区调度台方式组织页面主体。
- 登录页定位为“控制中心前厅”，有展示感但保持冷硬基调。
- 登录页采用“左侧品牌叙事区 + 右侧登录卡”结构，叙事以系统使命感为主。
- 顶部主导航采用“一级模块 + 少量全局状态块”的总控台式结构。
- 左侧工作台采用“摘要块”形态，标准顺序为当前视角、运行态势、关注事项、快速切换。
- 字体方向参考 Anthropic 的组合逻辑，但不照搬其商业字体，继续坚持中文优先和数据字独立强化。
- 字体策略正式选定为“无衬线主体系 + 小范围 serif + 独立数据字”的方案 B。
- Typography 的字号、字重、字距和行高规则已收敛到可执行范围。
- 第一批组件优先落地框架级组件与信息展示组件，再补交互基础与业务承载组件。
- 样板页开发顺序为：主框架样板 -> Dashboard 首页样板 -> 登录页样板 -> 基础组件固化。
