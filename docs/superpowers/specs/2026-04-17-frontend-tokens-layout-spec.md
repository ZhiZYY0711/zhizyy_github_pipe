# Frontend Tokens And Layout Spec

## 1. 文档定位

这份文档是已通过验收的原生样板的工程化落地规范，用来承接后续 Vue 3 重建工作。它不再讨论风格方向，而是明确：

- 哪些视觉决策已经固定
- 哪些值可以直接落到 CSS variables
- Dashboard 与登录页的页面结构、尺寸、层级和响应式约束

## 2. 适用范围

当前 tokens 和 layout spec 只覆盖第一阶段要落地的页面：

- 登录页
- 地图中心的 Dashboard 首页

明确不在本轮固定的内容：

- 表格页、表单页、抽屉、弹窗的完整组件体系
- 图表库接入细节
- 真正的数据接口契约
- 业务二级页的细分布局

## 3. 固定原则

### 3.1 页面级原则

- Dashboard 首页必须是单屏总览页，目标工作分辨率下不依赖页面纵向滚动。
- Dashboard 首页不再保留底部趋势区和辅助指标区，核心信息必须在首屏直接完成阅读。
- 地图是首页唯一主角，右侧只承担风险与处置态势，左侧只承担摘要型上下文，不承担长文案。
- 登录页保留左叙事右登录卡结构，但左侧以图案、图形和少量高密度文本为主，不做文字墙。

### 3.2 结构级原则

- 不使用传统左侧导航栏加右侧内容区的后台骨架。
- 顶部是一级模块与少量全局状态块，不承接局部筛选。
- 左侧是摘要块工作台，不是表单式筛选器。
- 模块之间必须有清晰拼接边界，不能做成一张边界模糊的大画布。

### 3.3 视觉级原则

- 主体是冷硬监控台，编辑感只做轻量渗透。
- 结构主要靠深浅表面层级建立，不靠重阴影和大面积发光。
- 强调色只服务状态、风险和关键操作，不做大面积背景。
- 中文优先，个性来自版式纪律、数据字、少量 serif 强调，不靠花字体。

## 4. Token 分层

推荐在 Vue 3 新工程中把 tokens 分成以下 6 层：

- `color`: 颜色与语义色
- `surface`: 背景、面板、覆盖层
- `type`: 字体、字号、字重、行高、字距
- `space`: 间距与模块缝隙
- `stroke`: 边线、分隔、描边强度
- `effect`: 阴影、渐变、特殊图形光晕

### 5.4 背景规则

- 页面背景必须保持深色矿物感，不允许切回白底后台。
- 背景允许使用极轻的青橙双向径向渐变，但透明度要压在 `0.04 - 0.07` 范围。
- 网格纹理只允许出现在大画布或叙事背景层，不能进入表单高频区。

## 6. Typography Tokens

### 6.1 字体栈

| Token | 推荐值 | 用途 |
| --- | --- | --- |
| `--font-sans` | `"Microsoft YaHei UI", "PingFang SC", "Noto Sans SC", sans-serif` | UI、正文、表单、标签 |
| `--font-serif` | `"Noto Serif SC", "Source Han Serif SC", "Songti SC", "STSong", serif` | 局部标题、仪式性标题、模块名强调 |
| `--font-mono` | `"JetBrains Mono", "IBM Plex Mono", "Consolas", "Courier New", monospace` | KPI、时间、编号、验证码、状态数字 |

说明：

- `sans` 是主体系，后续 Vue 3 实现默认全部先走 `sans`。
- `serif` 只允许用于登录页主标题、Dashboard 的少量模块标题、系统名等低频场景。
- `mono` 是控制台气质的重要来源，所有数据核心值统一走 `mono`。

### 6.2 字号层级

| Token | Value | 用途 |
| --- | --- | --- |
| `--text-display-lg` | `44px` | 登录页左侧主标题 |

| `--text-heading-lg` | `28px` | 登录卡主标题 |
| `--text-heading-md` | `24px` | 系统名、主区域标题 |
| `--text-section` | `16px` | 摘要块主标题 |
| `--text-body-md` | `15px` | 正文说明、说明句 |
| `--text-body-sm` | `13px` | 面板文字、告警描述、操作文案 |
| `--text-meta` | `12px` | 标签、辅助说明 |
| `--text-micro` | `10px` | eyebrow、弱标签、chip 标签 |
| `--text-data-lg` | `48px` | 登录页环形数字 |
| `--text-data-md` | `30px` | 左侧环形主数字 |
| `--text-data-sm` | `28px` | KPI 主数值 |
| `--text-data-xs` | `22px` | 验证码、次级数据块 |

### 6.3 行高与字距

| Token | Value | 用途 |
| --- | --- | --- |
| `--leading-display` | `1.15` | 大标题 |
| `--leading-body` | `1.6` | 默认正文 |
| `--leading-relaxed` | `1.7` | 登录页叙事辅助句 |
| `--tracking-wide` | `0.24em` | eyebrow |
| `--tracking-panel` | `0.12em` | 面板小标题 |
| `--tracking-button` | `0.14em` | 主按钮 |
| `--tracking-tight` | `0.04em` | 中文大标题 |

### 6.4 字重规则

- `400`: 常规说明、正文。
- `600`: 面板标题、状态标题、导航强调。
- `700`: 系统名、主按钮、KPI 主数值。
- 不建议使用超多字重层级，控制在 `400 / 600 / 700` 三档即可。

## 7. Space Tokens

推荐统一用 4 的倍数体系，样板中已经出现的稳定值如下：

| Token | Value | 用途 |
| --- | --- | --- |
| `--space-1` | `4px` | 微小内距 |
| `--space-2` | `8px` | 小标签、chip 内距 |
| `--space-3` | `12px` | 组件间基础缝隙 |
| `--space-4` | `16px` | 页面区块基础内边距 |
| `--space-5` | `18px` | 外框与窗口边缘的特殊安全边距 |
| `--space-6` | `20px` | 顶部栏左右内边距 |
| `--space-7` | `24px` | 登录卡核心区块间距 |
| `--space-8` | `32px` | 登录卡与大区块内边距 |
| `--space-9` | `40px` | 登录页右区整体留白 |
| `--space-10` | `72px` | 登录左叙事区大留白起点 |
| `--space-11` | `88px` | 登录页主内容顶部起始留白 |

执行规则：

- Dashboard 内部以 `12 / 16 / 18` 为主，不使用过大留白。
- 登录页允许 `24 / 32 / 40 / 72` 级别留白，以保留前厅感。
- 页面容器与浏览器边缘必须留出安全边距，不允许满贴屏幕。

## 8. Stroke, Radius, Effect Tokens

### 8.1 Radius

| Token | Value | 用途 |
| --- | --- | --- |
| `--radius-none` | `0` | 主体面板、卡片、导航块 |
| `--radius-round` | `999px` | 环形图案、圆点、状态粒子 |

规则：

- 结构面板默认硬边，不做软圆角后台。
- 圆形只用于图案、信号点、环形摘要，不用于主要面板卡片。

### 8.2 Shadow

| Token | Value | 用途 |
| --- | --- | --- |
| `--shadow-shell` | `0 20px 50px rgba(0, 0, 0, 0.42)` | Dashboard 外层框架 |
| `--shadow-card` | `0 28px 70px rgba(0, 0, 0, 0.42)` | 登录卡等高聚焦区 |

规则：

- 阴影只做深度感，不做漂浮感。
- 不允许给每个卡片都加夸张外阴影。

### 8.3 Effect

- 圆环、雷达、信号点允许使用低透明度同色光晕。
- 主按钮允许使用轻微暖色渐变。
- 地图主画面允许使用局部径向热区和路径线，但不能做炫技式发光动画。

## 9. Dashboard Layout Spec

## 9.1 页面骨架

Dashboard 必须遵守以下结构：

1. 外层页面安全边距
2. 单一主框架 `frame`
3. 顶部控制栏 `topbar`
4. 左侧摘要工作台 `left-rail`
5. 右侧主工作区 `main-stage`
6. 主工作区内部由 `kpi-strip + board` 组成
7. `board` 内部分为 `map-panel + side-stack`

### 9.2 尺寸基线

这些值已经在样板中验证过，可直接作为第一版实现基线：

| 区块 | 值 |
| --- | --- |
| 页面外层边距 | `18px` |
| 顶部栏高度 | `72px` |
| 顶部栏三列 | `290px / 1fr / 300px` |
| 左侧工作台宽度 | `260px` |
| 主区内边距 | `16px` |
| KPI 条最小高度 | `82px` |
| 主面板标题栏高度 | `48px` |
| 主画布与侧栈列比 | `1.6fr / 0.84fr` |

### 9.3 Dashboard 阅读顺序

阅读顺序必须固定为：

1. 顶部系统与全局状态
2. 左侧当前视角与态势摘要
3. 中央地图主画面
4. 右上重点风险
5. 右下处置进度

不允许把注意力先导向长列表或底部次要图表。

### 9.4 左侧工作台规则

左侧固定为 4 段：

1. `当前视角`
2. `运行态势`
3. `关注事项`
4. `快速切换`

执行约束：

- 左侧以图案型摘要为主，文本只保留短标签和关键值。
- 左侧总高度必须在目标分辨率下完整展示，不允许用户先滚左侧才能看完首屏结构。
- 如果后续接入真实数据导致内容变长，优先压缩文案和标签数量，不优先增加滚动。

### 9.5 KPI 条规则

- KPI 条仅保留 4 个压缩块。
- 每块只包含标签与主数字，不带长解释。
- 主数字统一使用 `mono`。
- 高风险和在线类指标用橙、青进行语义区分。

### 9.6 地图主画面规则

- 地图主画面必须占首页最大可视面积。
- 地图区允许自定义示意轮廓、路径线、热点环和气泡说明。
- 地图气泡文案必须短，使用三段结构：
  - 位置或对象
  - 核心判断
  - 1 到 2 个关键数据片段
- 说明性长文只能出现在原型注释阶段，正式实现需移除。

### 9.7 右侧态势区规则

右侧只保留 2 个堆叠面板：

- `重点风险`
- `处置进度`

执行约束：

- 风险面板优先于处置面板。
- 每个面板默认控制在 3 条以内，先保留高信号内容。
- 风险块允许语义色背景倾向，但透明度必须保持克制。
- 右侧不再加入排行、趋势、新闻式附属模块。

### 9.8 Dashboard 响应式约束

建议首版断点如下：

- `>= 1440px`: 保持样板原始双列结构。
- `1180px - 1439px`: 缩紧顶部栏和侧栏，但仍维持地图主导。
- `768px - 1179px`: 左侧摘要块移到地图上方，右侧两个面板下沉到地图下方。
- `< 768px`: Dashboard 不作为核心工作形态优化目标，只保证可读，不要求完整复刻桌面单屏体验。

补充规则：

- 桌面端必须优先服务单屏无页面滚动。
- 中窄屏可以允许区块纵向重排，但要避免元素互相覆盖。

## 10. Login Layout Spec

### 10.1 页面骨架

登录页固定为双区：

- 左区 `login-story`
- 右区 `login-access`

默认比例：

- `1.15fr / 0.85fr`

### 10.2 左区规则

左区不是营销海报，而是系统前厅。它由 5 类元素组成：

1. 深色叙事背景层
2. 网格纹理层
3. 环形/轨道/信号点图案层
4. 使命型标题与一句辅助说明
5. 一个图文混合的核心摘要模块

执行约束：

- 左区文案必须少。
- 主标题只保留一条使命句。
- 辅助说明只保留一条功能定义句。
- 剩余信息主要交给环形图、状态块、标签组表达。

### 10.3 右区规则

右区是纯操作入口，必须保持克制和直接：

- 单一登录卡即可
- 登录卡宽度控制在 `min(440px, 100%)`
- 登录卡内部以字段、验证码、主按钮、状态元信息组成
- 不增加装饰型副标题或多余插图

### 10.4 登录卡尺寸基线

| 区块 | 值 |
| --- | --- |
| 右区内边距 | `40px` |
| 登录卡宽度 | `min(440px, 100%)` |
| 登录卡内边距 | `34px 32px` |
| 输入框高度 | `46px` |
| 主按钮高度 | `48px` |
| 验证码区域宽度 | `112px` |

### 10.5 登录页字体与文案规则

- 左区主标题使用 `serif`。
- 登录卡标题使用 `sans`。
- 输入标签、辅助说明使用 `sans` 的 `12px - 13px`。
- 验证码和状态数字使用 `mono`。
- 文案语气必须正式、克制、带一点技术锋利感，不允许宣传口号化。

### 10.6 登录页响应式约束

- `>= 1180px`: 保持双栏。
- `< 1180px`: 退化成纵向堆叠，左叙事在上，登录卡在下。
- `< 720px`: 压缩大标题、环形尺寸和卡片内边距，保证可读和可点击。

## 11. 命名建议

后续 Vue 3 工程建议直接沿用以下结构命名，避免样板、设计稿、开发稿三套名字：

- `TopBar`
- `SystemBlock`
- `StatusBar`
- `LeftRail`
- `SummaryCard`
- `KpiStrip`
- `MapPanel`
- `RiskPanel`
- `TaskPanel`
- `LoginStory`
- `LoginAccess`
- `AccessCard`

- `--color-*`
- `--font-*`
- `--text-*`
- `--space-*`
- `--radius-*`
- `--shadow-*`

## 12. 哪些值是固定的，哪些值可调

### 固定项

- Dashboard 单屏、无底部趋势区
- 地图中心结构
- 左摘要、右风险处置的版式逻辑
- 登录页左右分区结构
- 深色冷硬材料感
- sans 主体系 + serif 点缀 + mono 数据字

### 可调项

- 同层级面板的具体宽度微调
- 个别模块的内边距和标签密度
- 图案层细节
- 响应式断点上的重排方式

## 13. 这份规范之后的正确顺序

完成这份 spec 之后，后续工作建议顺序如下：

1. 用本 spec 把 tokens 抽成 CSS variables 文件
2. 用本 spec 重建 Vue 3 的 Dashboard 与登录页静态骨架
3. 把页面骨架拆成框架级组件
4. 再开始接真实登录流和 Dashboard 数据

如果跳过这一步直接写 Vue 3 页面，后面大概率会再次出现样板与实现偏离的问题。
