# 总览大屏 Geo 性能优化 Spec

## 背景

总览页已经改成全屏态势地图。当前实现会按需加载行政区 GeoJSON，在缩放时动态合并可见下级边界，调用 ECharts `registerMap` 和 `setOption`，同时渲染管线、节点、告警等动态图层。

这次排查里看到性能问题主要分两类。第一类是 Geo 资源本身偏重：`frontend/public/geo` 目前有 3183 个 JSON 文件，未压缩约 274 MB，单个 `geo/index.json` 约 877 KB。第二类是本地资源已经加载后，交互仍然卡，因为缩放会在浏览器主线程里做 JSON 解析、下级 feature 合并、地图注册、ECharts 重绘和动态图层渲染。

现有 `frontend/scripts/build-geo-cdn-bundle.mjs` 方向是对的。它会生成 `frontend/dist/geo-cdn/v1`，只保留全国、省、市级 GeoJSON，并把大部分区县索引映射到市级文件。实际生成结果从原始 274 MB、3183 个 JSON，下降到约 27 MB、364 个 JSON。这会明显改善网络传输、CDN 缓存和请求数量，但不能单独消除地图渲染卡顿，因为浏览器仍然需要解析、注册和绘制加载到的 GeoJSON。

## 目标

第一个目标是让 Geo CDN bundle 达到生产可用。生成的 `index.json` 里每一个路径都必须指向 bundle 内真实存在的文件。直辖市区县等无法通过市级 GeoJSON 匹配的条目，也必须回退到 bundle 内的父级文件，而不是继续指向旧的 `/geo/admin/district/*.json`。

第二个目标是降低地图交互卡顿，并且不改变当前用户可见的数据语义。缩放和平移应该优先保持顺滑；下级边界加载、地图注册和重绘可以延后到交互结束后再做。大屏的视觉方向继续保留，但常态动画不能持续抢主线程预算。

第三个目标是把性能验收变成可测量结果。优化是否有效，不只看主观感觉，还要用 bundle 完整性、前端构建、浏览器控制台、Geo 请求和交互长任务采样来判断。

## 非目标

本 spec 不新增后端 API，不做数据库迁移，不引入真实管线几何绑定，也不替换 ECharts 或重做地图渲染引擎。

本 spec 不改变当前行政区钻取模型，也不移除现有 KPI、告警、tooltip、区域数据接口契约。

本 spec 不试图一次性解决所有后端 API 延迟。KPI、告警、监测聚合查询仍然是相关但独立的后端性能优化线。

## 方案设计

### Geo Bundle 完整性

继续由 `frontend/scripts/build-geo-cdn-bundle.mjs` 生成 CDN 目录 `frontend/dist/geo-cdn/v1`。文件模型保持精简：全国 GeoJSON、省级 GeoJSON、市级 GeoJSON 和一个生成后的索引文件。

区县条目需要通过确定性的父级回退链解析。优先映射到包含该区县的市级 GeoJSON，因为这是区县展示最合适的父级。若某个区县没有匹配到任何市级 GeoJSON，脚本应继续检查省级 GeoJSON，并把该区县映射到对应的省级 bundle 文件。若仍然无法匹配，脚本必须报错失败，不能静默输出 bundle 外路径。

这个变化会把 bundle 契约从“尽力生成”改为“闭环生成”：只要生成成功，`index.json` 就不允许引用未生成的文件。

### 地图明细加载

地图组件应该把动态下级边界当作交互后的增强内容。用户滚轮缩放和平移时，基础地图缩放应立即响应；下级边界获取、feature 合并和 `registerMap` 应在交互稳定后的 debounce 窗口再执行。

当前组件已经有明细加载 debounce，但阈值对重地图场景仍偏激进。应适当增加 debounce，避免连续滚轮过程中反复重建地图。这里明确优先保证交互顺滑，再让边界细节稍后补齐。

可见父级数量上限也应降低。全国视图展开城市边界时，一次拉取的省级子文件要更少；城市继续展开区县时，一次拉取的市级子文件也要更少。用户放大或钻取后仍能看到更细边界，但单次更新的 JSON 解析和渲染成本要有更低上限。

### 图层动画预算

管线图层保留视觉层级，但减少常态持续动画。图层开启时，所有管线仍然可见；流动动画只保留给选中管线、预警管线和严重管线。正常且未选中的管线不应持续运行 flow effect。

节点和告警强调继续保留，但预警、严重告警应优先于普通节点动画。如果第一轮优化后仍有明显卡顿，下一步再考虑把普通节点从 `effectScatter` 降级到非涟漪 scatter。

### CDN 运行契约

`VITE_GEO_CDN_BASE_URL=/geo-cdn/v1` 应继续配合 `resolveGeoAssetPath` 工作。生成的路径仍然以 `/geo/...` 为根，因为前端当前会去掉 `/geo/` 前缀后再拼接 CDN base。

生产 Nginx 可以继续把 `/geo-cdn/v1/` 代理到 OSS。启用 CDN 模式时，前端镜像不必包含完整原始 `public/geo` 目录，但部署流程必须先上传 `dist/geo-cdn/v1`，再启用 CDN base URL。

## 验收标准

生成的 CDN bundle 不能有悬空路径。测试应执行 `node scripts/build-geo-cdn-bundle.mjs`，读取 `dist/geo-cdn/v1/index.json`，并检查所有引用文件都存在于 `dist/geo-cdn/v1` 内。

生成 bundle 体积必须相对原始 Geo 数据有数量级改善。当前预期规模是约 27 MB、364 个 JSON，而不是原始 `frontend/public/geo` 的约 274 MB、3183 个 JSON。

`frontend` 下执行 `pnpm build` 必须通过。

浏览器采样 `/dashboard` 时，初始加载和重复滚轮交互不能出现控制台错误。第一轮优化的目标是减少缩放过程中的长任务数量和时长；此前观察到连续 zoom 会出现多次约 70-200 ms 的 long task，优化后应明显下降。具体帧间隔会受机器影响，所以以同一环境下的对比采样为准。

## 风险和取舍

将未匹配区县映射到省级文件，会减少资源量并避免 404，但某些直辖市区县钻取时可能展示父级几何，而不是独立区县 GeoJSON。对总览大屏来说这是可接受的，因为这里是指挥态势视图，不是精确 GIS 编辑器。

增加明细加载 debounce 会让边界细节出现得稍晚。这是有意取舍：交互响应优先，细节在用户停止操作后补齐。

减少正常管线动画会让画面少一些常态动效。这也是有意取舍：运营大屏里，动画应表达关注、选中或风险，而不是全场持续运动。

Geo bundle 能改善网络、缓存和请求风暴，但不能解决所有 ECharts 渲染成本，也不能解决后端查询延迟。如果这轮之后仍有明显慢感，下一条优化线应是合并总览页 summary API，并启用或修正后端 KPI、告警、区域监测数据的预聚合快路径。

## 验证计划

先做 bundle 完整性测试。当前脚本会失败，因为已经发现 116 个区县条目仍指向 `/geo/admin/district/*.json`，但这些文件不会生成到 `dist/geo-cdn/v1`。修复脚本后，同一个测试必须通过。

然后运行 `pnpm geo:bundle`，检查生成文件数量和体积。bundle 应只依赖 index、全国、省、市文件，不依赖 district 文件。

接着在 `frontend` 运行 `pnpm build`。

最后用 Playwright CLI 打开 `/dashboard`，使用开发 token 登录并采样重复滚轮交互。记录控制台错误、Geo 资源请求、canvas 数量、long task 和 frame delay，用来和本次排查得到的基线对比。
